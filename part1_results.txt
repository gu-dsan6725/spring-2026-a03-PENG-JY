Q2: What is the main entry point file for the registry service?
============================================================
The main entry point is `registry/main.py`.

This was confirmed by running `rg -n "FastAPI\(|uvicorn\.|__main__"` inside the `registry/` directory, which pointed exclusively to this file, and then reading it directly with `cat registry/main.py`. The file opens with:

```python
#!/usr/bin/env python3
"""
MCP Gateway Registry - Modern FastAPI Application

A clean, domain-driven FastAPI app for managing MCP (Model Context Protocol) servers.
This main.py file serves as the application coordinator, importing and registering
domain routers while handling core app configuration.
"""
```

It contains the `FastAPI()` app instantiation, lifespan context manager (startup/shutdown hooks), and all `include_router()` calls that wire in the domain route modules. No other file in the repository performs this role.

Q4: How does the authentication flow work, from token validation to user authorization?
============================================================
The system uses a two-layer architecture: a standalone `auth_server` handles token validation, and the `registry` application enforces per-endpoint authorization through FastAPI dependencies.

**Layer 1 — Auth Server validation (`auth_server/server.py`)**

Every request to the registry passes through Nginx, which forwards it to the auth server's `/validate` endpoint (line 1142) before proxying it onward.

The `validate_request` function (lines 1120–1205) runs the following checks in order:

1. **Extract headers** — reads `Authorization` (Bearer token) or falls back to `X-Authorization`; also extracts `X-User-Pool-Id`, `X-Client-Id`, `X-Region`, `X-Original-URL`, and `Cookie`
2. **Guard missing token** — raises `HTTPException(401)` if neither auth header is present
3. **Route to provider** — passes `X-User-Pool-Id` and `X-Client-Id` to `auth_server/providers/factory.py` (lines 20–47) which selects the correct `AuthProvider` implementation (Keycloak, Entra ID, or Cognito)
4. **Validate JWT** — calls the provider's `validate_token()` method, which fetches the JWKS endpoint and verifies the token signature and expiry
5. **Parse target** — extracts the server name and endpoint path from `X-Original-URL`; federation/peer requests receive a fast-path approval
6. **Scope check** — compares the token's scopes/groups against the permission requirements for the target endpoint (defined in `auth_server/scopes.yml`)
7. **Return result** — HTTP 200 with user info injected as headers (`X-User-Id`, `X-Username`, etc.) on success; HTTP 401/403 on failure

**Layer 2 — Registry authorization (`registry/auth/dependencies.py`)**

Protected route handlers declare a dependency on `nginx_proxied_auth` or `enhanced_auth`. These functions read the user info headers injected by Nginx and build a `UserContext` object, which the route handler uses to enforce ownership or role checks (e.g. only the server owner may call `/edit/*`).

Key files: `auth_server/server.py`, `auth_server/providers/base.py`, `auth_server/providers/factory.py`, `auth_server/providers/keycloak.py`, `auth_server/providers/entra.py`, `registry/auth/dependencies.py`, `registry/auth/routes.py`

Q6: How would you add support for a new OAuth provider (e.g., Okta)?
============================================================
The auth system uses a clean provider pattern. All provider implementations live in `auth_server/providers/` and inherit from a single abstract base class. Adding Okta requires changes to exactly four locations:

**Step 1 — Create `auth_server/providers/okta.py`**

Subclass `AuthProvider` from `auth_server/providers/base.py` (lines 15–174) and implement all 8 abstract methods:

| Method | Line range | What it must do |
|--------|-----------|----------------|
| `validate_token` | 19–43 | Verify token signature via JWKS, return user info dict |
| `get_jwks` | 46–54 | Fetch Okta's JWKS endpoint for key material |
| `exchange_code_for_token` | 57–80 | POST to Okta token endpoint with auth code |
| `get_user_info` | 83–102 | Call Okta `/userinfo` with the access token |
| `get_auth_url` | 105–122 | Build the Okta authorization redirect URL |
| `get_logout_url` | 125–137 | Build the Okta end-session URL |
| `refresh_token` | 140–155 | Exchange refresh token for new access token |
| `validate_m2m_token` | 158–173 | Validate client-credentials tokens for service accounts |

**Step 2 — Register in `auth_server/providers/factory.py` (lines 20–47)**

Add a new branch to `get_auth_provider()` for `provider_type == "okta"` that instantiates `OktaProvider` with config drawn from environment variables. The existing Keycloak and Entra branches show the expected pattern.

**Step 3 — Add config to `auth_server/oauth2_providers.yml`**

Add an `okta` block with at minimum: `issuer_url`, `client_id`, `client_secret`, `scopes`. Mirror the structure of the existing `keycloak` or `entra` entries in this file.

**Step 4 — Update `api/registry_management.py`**

The CLI management tool references provider names at many locations (confirmed lines include 172, 381, 523, 527, 543, 544, 550, 591, 954, 977, 979, 993, 996, 999, 1546, 1636, 1688–1702, 1733, 2704, 2741, 2763, 3886, 3917, 4065). Specifically:
- Add `"okta"` to the `provider_map` dictionary
- Add Okta-aware branches inside `create_keycloak_group`, `delete_keycloak_group`, and `list_keycloak_iam_groups` (or refactor these into provider-agnostic functions)
- Update the `--keycloak-url` CLI option to accept an Okta domain equivalent

**Required new environment variables** (following `.env.example` conventions):
- `OKTA_ISSUER_URL` — e.g. `https://your-domain.okta.com/oauth2/default`
- `OKTA_CLIENT_ID`
- `OKTA_CLIENT_SECRET`
- `OKTA_AUDIENCE`

Files to modify: `auth_server/providers/factory.py`, `auth_server/oauth2_providers.yml`, `api/registry_management.py`
Files to create: `auth_server/providers/okta.py`
